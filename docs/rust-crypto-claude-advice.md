# Поради для CLAUDE.md — Rust криптографічна бібліотека

> **Статус: розподілено.** Профільний крипто-контент із Частини 1 перенесено в `SECURITY.md`
> (hard constraints, threat model, supply-chain) та `DECISIONS.md` (D-01…D-06). Дисципліна агента
> (правило трьох спроб, дослідження перед реалізацією, недовіра "зеленим" тестам) — у `CLAUDE.md`
> ("Agent discipline"). Частина 2 (економія контексту) уже покрита harness-практиками з
> `~/.claude/CLAUDE.md` і глобальними інструкціями — окремо не дублюється. Цей файл лишається як
> оригінальне джерело/обґрунтування, не як canonical-документ для читання наперед.

Витягнуто з досвіду проєкту Pakko (Windows-архіватор, C#/C++). Частина 1 — інженерні
конвенції, які варто перенести/адаптувати. Частина 2 — як Claude Code повинен читати файли
та вносити зміни, щоб економити контекст/токени.

---

## Частина 1 — Структура документації та інженерна дисципліна

### Документація

- **Один файл-індекс (`CLAUDE.md`)**, що перелічує кожен інший `.md`-файл з колонками
  "Read when / Update when" і **canonical owner** для кожної теми. Якщо тема вже описана
  в одному файлі — інші файли лише лінкують на нього, ніколи не дублюють таблицю.
- **`DECISIONS.md`** — архітектурні рішення *разом із відхиленими альтернативами та причиною
  відхилення*. Для крипто-бібліотеки: чому обрано конкретний набір primitives/curves, чому не
  підтримується legacy-режим, чому залежність прийнята чи відхилена. Дописувати в момент
  рішення, не заднім числом.
- **`SECURITY.md`** — модель загроз, explicit out-of-scope, таблиця supply-chain оцінки
  залежностей (розробник, reproducible builds, незалежний аудит, CVE-історія) — застосовувати
  до кожного crypto-крейта перед додаванням. Розділ "Reporting Vulnerabilities" — приватне
  розкриття (GitHub Security Advisories), ніколи публічний issue.
- **`TASKS.md` / `TASKS_DONE.md`** — наскрізна нумерація задач (`T-xx`), acceptance criteria
  на кожну. Задача переходить у "done" НЕ тому що `cargo test` зелений — потрібна окрема
  верифікація (для крипто: crossverification проти тестових векторів + незалежної реалізації).
- **`CHANGELOG.md`** — секція на реліз, пишеться в момент релізу зі списком задач з
  попереднього тегу.
- **"Known test gaps" секція** — документувати flaky-тест і правило "один реран ізольовано
  перед тим як вважати регресією", замість мовчазного ігнорування або постійних
  перезапусків без розбору.

### Дисципліна агента

- **Правило трьох спроб**: та сама проблема не вирішується після 3 різних підходів —
  зупинитись, доповісти що спробовано і що невідомо, чекати вказівок. Не пробувати 4-й
  підхід самовільно. Особливо для toolchain/build/CI проблем.
- **Дослідження перед реалізацією**: жоден primitive не пишеться "з пам'яті" — звірятись з
  першоджерелом (конкретний розділ RFC/NIST-документа, реальний код референсної реалізації),
  а не з переказом. Фіксувати посилання в `DECISIONS.md`.
- **Не довіряти "тести зелені" для security-critical коду.** Власна реалізація має звірятись
  з тестовими векторами (NIST CAVP/RFC) **і** з незалежним крейтом (`ring`, RustCrypto) — не
  лише self-consistency. Баг може пройти повз власні тести, але провалитись на незалежному
  ридері/реалізації.
- **Грепати весь репозиторій, не лише план**, перед зміною публічного API — план міг бути
  написаний до появи нового споживача цього API.
- Мінімальні дифи, без спекулятивних абстракцій. Коментарі лише WHY, коли неочевидно
  (workaround, side-channel причина, invariant), ніколи WHAT.

### Специфічно для крипто-бібліотеки

- Explicit hard-constraints список: "жоден primitive не пишеться без цитати конкретного
  розділу специфікації", "жодної secret-dependent гілки/індексації масиву", "усі порівняння
  секретів через `subtle::ConstantTimeEq`, ніколи `==`", "усі типи ключового матеріалу —
  `Zeroize`/`ZeroizeOnDrop`", "жодних логів секретного матеріалу", "жодних власних
  crypto-примітивів, вигаданих з нуля".
- **`unsafe`-політика**: `unsafe` тільки в ізольованих, окремо перевірених модулях, з
  коментарем що саме invariant гарантує безпеку. `cargo miri test` — обов'язковий шар для
  UB-детекції (аналог незалежної WACK-перевірки в Pakko: інструмент ловить те, що звичайні
  тести пропускають).
- **Бенчмарк-методологія**: ratio-based порівняння на одній машині, в одному прогоні, проти
  еталонної реалізації (`criterion` + порівняння з `ring`/OpenSSL в тому самому бенчі), з
  толерансом (напр. 3x) — НЕ абсолютний поріг часу. Це єдиний підхід, що узагальнюється на
  довільну машину (підтверджено дослідженням BenchmarkDotNet/criterion.rs/benchstat).
- **Fuzzing — обов'язковий шар**, не опційний: `cargo fuzz` для кожного парсера вхідних
  байтів (DER/ASN.1, message formats).
- **Supply-chain перевірка** кожного crypto-крейта перед додаванням залежності —
  та сама таблиця що в `SECURITY.md` (developer, reproducible builds, аудит, CVE-історія).

## Частина 2 — Як читати файли й вносити зміни, щоб економити контекст

Ці практики напряму зменшують кількість токенів/викликів інструментів за сесію —
критично для довгих сесій і для вартості (API/ключі).

- **Пошук замість повного читання.** Для пошуку символу/шаблону — `Grep`/`rg`, не читання
  цілих файлів "на око". Для пошуку файлів за іменем/шаблоном — `Glob`, не рекурсивний
  листинг директорії.
- **Читати файл цілим лише коли треба.** Якщо відомо, яка частина файлу потрібна (напр. після
  Grep дав номер рядка) — читати з `offset`/`limit`, а не весь файл, особливо для великих
  логів/згенерованого коду.
- **Не перечитувати файл одразу після Edit/Write "для перевірки".** Інструмент Edit сам
  видасть помилку, якщо заміна не відбулась (`old_string` не знайдено) — успішний виклик уже
  є доказом, що зміна застосувалась. Повторне читання — витрачений виклик.
  Виняток: коли потрібно перевірити *результат виконання* коду (тест, компіляція), а не сам
  факт запису файлу — тоді довіряти exit-коду/виводу команди, а не власному "eyeball"-читанню.
  Compiler/test exit status — авторитетне джерело правди про коректність, не повторне читання
  людиною (чи агентом) вихідного файлу.
  ```
  Скорочено: Edit --> (успіх) --> НЕ перечитувати файл --> продовжувати.
  Edit --> помилка --> прочитати актуальний вміст --> зрозуміти чому не збіглось --> повторити Edit.
  ```
- **`Edit`, не `Write`, для існуючих файлів.** `Edit` передає лише діф (old_string/new_string),
  `Write` вимагає повного вмісту файлу в запиті — на великому файлі це на порядки дорожче за
  токенами. `Write` — тільки для нових файлів або справді повного переписування "з нуля" на
  явний запит.
- **Групувати незалежні виклики інструментів в одному ході.** Кілька `Read`/`Grep` без
  залежностей одне від одного — паралельно в одному повідомленні, не послідовно по одному —
  економить не токени напряму, а раунд-трипи/латентність, але зменшує і кількість
  проміжних системних повідомлень, які накопичуються в контексті.
  Послідовно — тільки коли результат одного виклику потрібен як вхід для наступного.
  Правило: незалежні → паралельно; залежні → послідовно, ніколи навпаки.
- **Форкати/делегувати "дослідницьку" роботу, чий сирий вивід не знадобиться пізніше.** Якщо
  завдання — прочитати купу файлів і повернути висновок (не самі файли), краще запустити
  форк/сабагента, який поверне стислий підсумок, ніж тягнути весь сирий вивід (сотні рядків
  логів, дифів, деревʼя каталогів) у головний контекст.
- **Не дампити цілі великі логи/вивід збірки.** Фільтрувати релевантні рядки (Grep по
  ключовому слову помилки) замість виводу всього `cargo build`/`cargo test` stdout, коли
  файл великий і відома природа помилки.
- **Не тримати в контексті документи, що вже застаріли/не потрібні для поточного кроку.**
  Для дуже великого файлу рішень (`DECISIONS.md`, що постійно росте) — grep по заголовку
  конкретної секції замість повного перечитування файлу щоразу.
- **Завантажувати інструменти під задачу, не всі одразу**, коли інструменти доступні через
  відкладене завантаження (search-by-name) — запитувати весь потрібний набір одним викликом
  наперед, а не по одному з кожною новою підзадачею.
