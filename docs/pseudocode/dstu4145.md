# DSTU 4145-2002 — pseudocode

No official standard text or designers' paper for DSTU 4145 exists in `docs/papers/` (see
`ORACLES.md`) — this pseudocode is transcribed instead from
`oracles/bouncycastle-java/core/src/main/java/org/bouncycastle/crypto/signers/DSTU4145Signer.java`
(MIT licensed, audited, decades in production — the primary oracle for this algorithm per
`ORACLES.md`). **Different provenance than the Kalyna/Kupyna/Strumok pseudocode docs**: those are
spec-transcriptions cross-checked against code; this is a code-transcription, because no spec
text is available to transcribe from. Per `DECISIONS.md` D-02, DSTU 4145 is wrapped from Bouncy
Castle for Java/.NET rather than reimplemented, and used as a second verification oracle for a
Rust port — so a direct, faithful reading of its logic is the right starting point here, not a
spec restatement.

Curve is over a binary field GF(2^m) (not the twisted-Edwards curves of DSTU 9041 — different
algorithm, see the algorithms table in `CLAUDE.md`). Domain parameters: curve `E`, base point `G`
of prime order `n`, private key `d` (integer, `0 < d < n`), public key `Q = d·G`.

## Message-hash preprocessing (`hash2FieldElement`)

The signer takes an already-computed message hash `H` (the digest algorithm itself is chosen by
a wrapping layer, not by this class — DSTU 4145 predates Kupyna, so the historical standard hash
is GOST 34.311; which digest to actually pair it with in this project needs a separate decision,
not assumed here):

```
hash2FieldElement(H):
    data ← reverse_bytes(H)                          // byte-order flip
    x    ← truncate(data as unsigned big-endian integer, field_size_in_bits)
    return x as a field element of GF(2^m)
```

`truncate(x, bits)`: if `x` needs more than `bits` bits to represent, reduce `x mod 2^bits`
(keep the low `bits` bits); otherwise leave unchanged.

## Signing (`generateSignature`)

```
generateSignature(H, d, G, n, curve):
    h ← hash2FieldElement(curve, H)
    if h == 0: h ← 1                      // field element one, avoids a degenerate hash

    loop:
        loop:
            loop:
                e ← random integer in [1, n-1]
                F_e ← x-coordinate of (e · G), affine
            while F_e == 0
            y ← h * F_e                    // field multiplication
            r ← truncate(y as integer, n.bit_length() - 1)
        while r == 0

        s ← (r * d + e) mod n
    while s == 0

    return (r, s)
```

## Verification (`verifySignature`)

```
verifySignature(H, r, s, Q, G, n, curve):
    if r <= 0 or s <= 0: return false
    if r >= n or s >= n: return false

    h ← hash2FieldElement(curve, H)
    if h == 0: h ← 1

    R ← (s · G) + (r · Q), affine           // combined multiply-and-add
    if R is the point at infinity: return false

    y ← h * (x-coordinate of R)
    r' ← truncate(y as integer, n.bit_length() - 1)
    return r' == r
```

## Notes on fidelity to this oracle

- `s·G + r·Q` is computed via `ECAlgorithms.sumOfTwoMultiplies` (Shamir's trick) in the oracle —
  a performance optimization, mathematically equivalent to two separate scalar multiplications
  added together. Any correct EC point arithmetic gives the same result; this is not part of the
  algorithm's definition, just how the oracle happens to compute it fast.
- Signing's inner-loop base-point multiplication (`e · G`) uses `FixedPointCombMultiplier` in the
  oracle, another performance choice (precomputed comb tables for a fixed point), not part of the
  algorithm.
- The triple nested retry loop (`e`/`F_e` degenerate, `r` zero, `s` zero) exists because DSTU 4145
  signing, like DSA/ECDSA, must reject a vanishingly rare set of unlucky random choices that would
  produce a value the verifier can't distinguish from zero or that would leak `d`. It is not an
  optional robustness addition — an implementation that skips these checks is not the same
  algorithm.

## What is deliberately not restated here

Curve domain parameter selection (`oracles/bouncycastle-java/.../DSTU4145NamedCurves.java`), the
compressed/precomputed point encoding used for interoperable signature/key serialization
(`DSTU4145PointEncoder.java`, `DSTU4145ECBinary.java`), and key-pair generation
(`DSTU4145KeyPairGenerator.java`) are separate concerns from the sign/verify core above — read
those files directly when the time comes to implement domain-parameter handling or wire encoding,
rather than through a paraphrase here.
